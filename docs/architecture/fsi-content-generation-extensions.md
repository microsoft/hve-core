> [!NOTE]
> This is a **draft proposal**, not an accepted design. It exists to focus discussion on a concrete set of extensions before any schema, discovery module, or host agent changes land. Nothing in this document is implemented today.

## Recommendation at a glance

**Go.** Prototype Extensions 2 (`itemKind: document-section`) and 3 (variable substitution) first against a single content type (PRD), then land Extensions 1, 4, 5, and 6+8 in sequence. Defer Extension 7 until evidence demands it.

The eight gaps are real, the extensions are additive, schema-compatible, and host-neutral, and the consumer set is already plural: PRDs, ADRs, BRDs, RCAs, user journeys, RAI plans, security plans, SSSC plans, and engineering-fundamentals briefs all live in `docs/templates/` today as Markdown files that humans hand-fill. Each is a candidate document-section host. The PRD builder is the first instance, not the only one.

The earlier framing of "single host risk" does not survive contact with the existing template inventory. The strategic question is no longer "will a second host appear?" but "in what order do we onboard the existing templates?"

The full reasoning, sized effort, and rejected alternatives are in [Alternatives considered](#alternatives-considered) and [Effort and risk](#effort-and-risk) below.

## Known consumers

The following document types already exist as templates in `docs/templates/` and are direct candidates for the `document-section` itemKind. Each one is a hand-filled Markdown file today, with its own implicit section list, prompts, and variables that are reinvented per author.

| Template                              | Today                         | What FSI extensions enable                                                        |
|---------------------------------------|-------------------------------|-----------------------------------------------------------------------------------|
| `prd-template` (planned)              | Not yet authored              | First prototype target                                                            |
| `adr-template-solutions.md`           | Static Markdown               | Section sequencing, decision-context variables, governance review cadence         |
| `brd-template.md`                     | Static Markdown               | Stakeholder/scope variables, regulated vs. standard variants via `selectWhen`     |
| `rca-template.md`                     | Static Markdown               | Incident-context variables, revisitable phases (post-mortem iteration)            |
| `user-journey-template.md`            | Static Markdown               | Persona variables, optional sections per journey type                             |
| `rai-plan-template.md`                | Static Markdown               | Already governed by an FSI-adjacent agent; aligns with planner contract           |
| `security-plan-template.md`           | Static Markdown               | Same as above                                                                     |
| `sssc-plan-template.md`               | Static Markdown               | Same as above                                                                     |
| `engineering-fundamentals.md`         | Static Markdown               | Style-guide reference, governance ownership                                       |
| `security-plan-template.md` (planner) | Generated by Security Planner | Override-and-sign: planner output replayable as a signed Framework Skill artifact |
| `rai-plan-template.md` (planner)      | Generated by RAI Planner      | Same as above; signed plan attestations bound to inputs                           |
| `sssc-plan-template.md` (planner)     | Generated by SSSC Planner     | Same as above; pairs naturally with the existing Sigstore skill                   |

The planner row deserves a callout. SSSC, RAI, and Security planners already produce structured documents today (they just emit them through host-specific code paths). If those outputs are expressed as `document-section` items with declared inputs and a `governance:` block, the same signing pipeline (see Extension 9) covers them. The planners do not have to migrate to be useful, but the contract is wide enough to receive them when they want in.

This inventory is the evidence that the proposal serves a class of consumers, not a single host. It also dictates the prototype order: PRD first because it is greenfield (no migration cost), ADR/BRD next because they are the highest-volume authoring surfaces, RCA after because revisitable phases (Extension 7) get their first real consumer there.

## Why this proposal exists

The Framework Skill contract was shaped by two consumer classes: planning agents (SSSC, RAI) and reviewer agents (Code Review Standards, dependency-reviewer, doc-update-checker). Both walk a finite, ordered list of items per phase and emit findings or work items. The contract fits them well.

A third consumer class is now visible on the roadmap: **content generators**. The PRD builder is the first concrete instance. It walks a sequence of authoring stages, prompts the user for inputs, renders sections of a document, and ultimately produces an artifact (Markdown, then `.docx`) that a human edits.

A reasonable user request is "let me bring my own PRD template as a Framework Skill, and chain a Word converter skill on the back." When we project the current FSI contract onto that workflow, eight gaps surface.

This document proposes minimal, additive extensions that close those gaps while preserving the properties that make Framework Skills useful today: one manifest schema, one discovery module, one validation cmdlet, host-owned phase semantics, and BYOF parity between built-in and user-authored Framework Skills.

## Design constraints

Every proposal in this document is held to four rules:

1. **Additive only.** Existing manifests and item files MUST remain valid. New fields are optional; defaults preserve current behavior.
2. **One schema, many `itemKind` values.** The manifest schema does not learn about content generation. New per-item shapes ride on new `itemKind` values with their own per-item schemas under `scripts/linting/schemas/`.
3. **Host owns meaning; Framework Skill owns shape.** New extension points must not require the manifest schema to enumerate host-specific vocabulary (phase names, variable names, output formats).
4. **No silent shadowing.** Anything a user-authored Framework Skill can do, a built-in Framework Skill can do, and vice versa.

A proposal that violates any of these is rejected on principle, regardless of how convenient it would be.

## The eight gaps

The fit-gap analysis identified these blockers when projecting an FSI-shaped PRD builder onto today's contract. A ninth gap (variable persistence across runs) surfaced during review and is addressed by Extension 8 below.

| # | Gap                                                             | Why it blocks content generation                                                              |
|---|-----------------------------------------------------------------|-----------------------------------------------------------------------------------------------|
| 1 | No skill-chaining or pipeline contract                          | Renderer skill cannot hand its Markdown to a `.docx` converter skill in a declared sequence.  |
| 2 | `itemKind` does not model document sections                     | A PRD section is not a control, criterion, or rule; per-item validation has nowhere to bind.  |
| 3 | No template-rendering or variable-substitution contract         | Section prompts cannot declare the inputs they need or the slots a renderer must fill.        |
| 4 | Binary artifact outputs (`.docx`) sit outside skill conventions | Current skills emit text; there is no contract for skills that write or consume binary files. |
| 5 | `applicability` is awkward for content routing                  | Choosing which template variant to render is selection logic, not exclusion logic.            |
| 6 | No content-governance metadata                                  | Owner, review cadence, deprecation, and style-guide pointers are free-form today.             |
| 7 | Phase model is one-pass; PRD authoring is non-linear            | Authors revisit Background after drafting Success Metrics; phases assume forward motion.      |
| 8 | Free-form `metadata` is too weak for governance and provenance  | Schema cannot enforce ownership, review state, or deprecation policy.                         |

The remainder of this document proposes one extension per gap, in the order most likely to be implemented.

## Extension 1: Pipeline manifests

> Closes Gap 1 (skill chaining). Enables Gap 4 (binary outputs).

### Problem

A PRD builder host wants to declare: "render Markdown using `prd-renderer`, then convert to `.docx` using `markdown-to-docx`." Today, the consuming agent has to encode that sequence in its own prompt. A user who swaps the renderer or the converter has to fork the agent.

### Proposal

Introduce an optional `pipeline:` block on the manifest. It declares an ordered list of named stages, each with an input contract (artifact kind it consumes) and an output contract (artifact kind it produces). The manifest does not name skill files; it names **stage roles** that the host resolves to skills via its own configuration.

```yaml
framework: my-prd-template
version: "2026.1"
domain: product-planning
itemKind: document-section
status: draft
pipeline:
  - stage: render
    consumes: section-prompts
    produces: markdown
  - stage: convert
    consumes: markdown
    produces: docx
phaseMap:
  outline:
    - background
    - problem-statement
  draft:
    - success-metrics
    - non-goals
```

The host agent reads `pipeline:`, resolves each `stage:` to a skill in its own registry (defaults shipped, user overrides honored), and runs the stages in order. Each stage receives the previous stage's output by `consumes`/`produces` contract.

### Schema impact

`framework-skill-manifest.schema.json` gains an optional `pipeline` array. Each entry requires `stage`, `consumes`, `produces`. `stage` is a lower-kebab string. `consumes` and `produces` are strings drawn from a host-owned registry; the schema does not enumerate them, mirroring the `phaseMap` design choice.

### Open questions

* Does the schema require `consumes` of stage N+1 to equal `produces` of stage N? Strict checking is friendlier; loose checking lets hosts insert implicit adapters.
* Should `pipeline:` allow branching (multiple consumers of one output)? A linear list covers the PRD case; branching is YAGNI until a second host needs it.

## Extension 2: `itemKind: document-section`

> Closes Gap 2. Depends on Extension 3 for variable contracts.

### Problem

Today the recognized `itemKind` values are `control`, `criterion`, `rule`, `principle`, `capability`. A PRD section ("Background", "Success Metrics") fits none. Per-item validation has no schema to bind to, so authors either misappropriate `criterion` (and confuse reviewers) or skip per-item validation entirely.

### Proposal

Recognize a new `itemKind: document-section`. Per-item files validate against a new `document-section.schema.json` under `scripts/linting/schemas/`. The schema captures what a renderer needs:

```yaml
# items/background.yml
id: background
title: Background
order: 10
required: true
prompt: |
  Describe the user problem, the population it affects, and the cost of the
  status quo. Use {{product_name}} when referring to the product.
inputs:
  - name: product_name
    kind: string
    required: true
  - name: target_users
    kind: string-list
    required: true
output:
  format: markdown
  heading_level: 2
  max_words: 400
selectWhen:
  discriminator: prd_template_variant
  values: [standard, regulated]
```

`required`, `order`, `prompt`, `inputs`, `output`, and `selectWhen` are all per-item fields owned by the new schema. The manifest schema does not change.

### Schema impact

* New file: `scripts/linting/schemas/document-section.schema.json`.
* `npm run validate:skills` learns to dispatch on `itemKind` and apply the matching per-item schema.

### Open questions

* Should `inputs[].kind` be an enumerated set (`string`, `string-list`, `enum`, `multiline`, `boolean`, `number`) or extensible per host? Enumerated is safer for tooling.
* Does `output.format` overlap with Extension 1's `produces`? They are at different layers (per-item vs. pipeline-stage) but the vocabulary should be aligned.

## Extension 3: Variable-substitution contract

> Closes Gap 3. Required by Extension 2.

### Problem

The renderer needs to know which variables a section consumes, where they appear in the prompt, and how to validate the user-supplied values. There is no shared syntax today, which means every renderer reinvents one.

### Proposal

* **Substitution syntax:** `{{variable_name}}` (double-brace, snake_case). The choice matches Mustache, Liquid, and existing prompt-template conventions in `.github/instructions/` files. No conditionals or loops at the contract layer; complexity belongs in the renderer skill if the host needs it.
* **Declaration:** every variable referenced in `prompt:` MUST appear in the section's `inputs:` list. Validation enforces this.
* **Collection:** the host agent collects values for each declared input before invoking the renderer stage, using the section's `inputs[].kind` to drive its own UI.
* **Scoping:** variables are resolved per-section unless the manifest declares a `globals:` block of variables collected once and reused across all sections.

```yaml
# manifest excerpt
globals:
  - name: product_name
    kind: string
    required: true
  - name: target_release
    kind: string
    required: false
```

### Schema impact

* `framework-skill-manifest.schema.json` gains an optional `globals` array using the same shape as the per-item `inputs`.
* The per-item `document-section.schema.json` declares the `inputs` shape.

### Open questions

* Should `validate:skills` lint that every `{{var}}` in `prompt:` resolves against `inputs` ∪ `globals`? Strongly yes; it's a free correctness win.
* Do globals need an "ask-once-per-session" vs. "ask-per-document" semantic? Probably "per-document"; per-session is a host concern.

## Extension 4: Binary artifact contract

> Closes Gap 4. Builds on Extension 1.

### Problem

Today's skills emit text (analysis, findings, work items). A `markdown-to-docx` skill emits a binary file. There is no shared convention for where it lands, how the next stage discovers it, or how validation handles it.

### Proposal

* Each pipeline stage declares its `produces:` as a content kind. Binary kinds are namespaced (`binary/docx`, `binary/pdf`, `binary/pptx`) to keep them distinguishable from text kinds (`markdown`, `json`, `yaml`).
* Pipeline stages exchange artifacts through a host-managed working directory under `.copilot-tracking/<host-domain>/<run-id>/`. Stage N writes its output there; stage N+1 reads from the same path. The stage contract is "produce the artifact at the path the host gives you, return its relative path."
* Validation tooling treats binary artifacts as opaque: it confirms the file exists, has non-zero size, and matches the declared kind by extension or magic bytes. It does not parse the content.

### Schema impact

* No manifest schema changes beyond Extension 1.
* `validate:skills` learns to recognize binary `produces` kinds and apply the file-existence check.
* A new shared instruction file under `.github/instructions/shared/` documents the working-directory convention so any host can adopt it.

### Open questions

* Should artifacts persist across runs (cache) or be ephemeral? Default ephemeral; opt-in cache later.
* How are binary artifacts surfaced to the user when the host completes? Likely a copy to a user-visible path declared by the host, not by the Framework Skill.

## Extension 5: First-class `selectWhen` predicate

> Closes Gap 5.

### Problem

`applicability` is shaped for "this control is structurally out of scope here." It uses `naWhen` and produces an `n/a` score. Selecting between two PRD template variants ("standard" vs. "regulated") is a positive choice, not an exclusion. Forcing it through `applicability` means every variant is `n/a` for every other variant, which is true but useless.

### Proposal

Introduce a per-item `selectWhen:` predicate (visible in the Extension 2 example). It mirrors `applicability.naWhen` shape:

```yaml
selectWhen:
  discriminator: prd_template_variant
  values: [regulated]
```

The host evaluates `selectWhen` against `state.projectContext` (or its content-generation equivalent) and **includes** the item only when the discriminator value matches one of the listed values. Items without `selectWhen` are always included.

### Schema impact

* The per-item `document-section.schema.json` declares `selectWhen`.
* `applicability` and `selectWhen` are explicitly distinct fields. An item MAY declare both: `applicability` answers "is this in scope at all?" and `selectWhen` answers "given it is in scope, do we want it for this variant?"

### Open questions

* Should `selectWhen` accept negation (`notValues:`)? Probably; symmetry with `naWhen` reads cleanly.
* Does `selectWhen` belong only on `document-section`, or is it general enough for `control` and `rule`? Likely general; defer the decision until a second host needs it.

## Extension 6 + 8: Governance metadata block

> Closes Gaps 6 and 8 together. The two are the same problem at different fidelity.

### Problem

Today's `metadata` is free-form. Useful in early imports, weak when an org needs to enforce review cadence, ownership, deprecation, or style-guide pointers. The PRD use case makes this acute: a stale PRD template causes silent harm.

### Proposal

Introduce an optional `governance:` block on the manifest with a structured schema. Free-form `metadata` remains for everything else.

```yaml
governance:
  owners:
    - team: product-platform
      contact: product-platform@example.com
  review:
    cadence: quarterly
    last_reviewed: "2026-04-01"
    next_review_due: "2026-07-01"
  approval:
    approved_by: product-platform-leadership
    approved_at: "2026-04-01T15:00:00Z"
  deprecation:
    deprecated: false
  style_guide: https://example.com/style-guide
```

Validation:

* `governance` is optional. When present, every sub-field is validated against the schema.
* `validate:skills` warns when `next_review_due` is in the past.
* `validate:skills` warns when `deprecation.deprecated: true` and any host references the Framework Skill without setting an "include deprecated" flag.
* Hosts SHOULD surface `governance.deprecation.deprecated` and `governance.review.next_review_due` to users when loading the Framework Skill.

### Schema impact

* `framework-skill-manifest.schema.json` gains an optional `governance` object with strictly typed fields.
* `metadata` retains its free-form shape and is no longer the recommended home for governance signals.

### Open questions

* Migration path for existing Framework Skills that put `owners` or `review_required` under `metadata`? Soft-deprecate via warnings; do not break.
* Is `governance` overkill for a small built-in Framework Skill like `sigstore`? It is optional; small Framework Skills ignore it.

## Extension 7: Non-linear phase mode

> Closes Gap 7. Lowest priority; highest design risk.

### Problem

`phaseMap` keys are ordered, and the host walks them forward. PRD authoring is iterative: the author finishes Success Metrics and discovers the Background section needs revision. Today's contract assumes forward motion.

### Proposal

Add an optional manifest field `phaseMode:` with values `sequential` (default, current behavior) and `revisitable`. When `revisitable`, the host MAY re-enter completed phases without resetting state. Per-item `revisitable: true` is also legal for finer control.

```yaml
phaseMode: revisitable
phaseMap:
  outline:
    - background
    - problem-statement
  draft:
    - success-metrics
```

The host owns what "revisit" means in its own UX. The Framework Skill declares only that revisits are legal.

### Schema impact

* Manifest gains optional `phaseMode` enum.
* Per-item schemas MAY declare `revisitable: true`.

### Open questions

* Does this leak host concerns into the Framework Skill? The flag is so thin (it just says "I do not assume one-pass") that it stays Framework-Skill-shaped.
* Is `phaseMode: revisitable` useful for any non-content-generation host? Possibly DT method coaches; verify before promoting from draft.

## Extension 8: Variable persistence across runs

> Closes a gap surfaced after the initial eight: variable values are re-collected from scratch on every run.

### Problem

Extension 3 declares the contract for variables but says nothing about their lifetime. As written, every PRD, ADR, and BRD run prompts the user for `product_name`, `team_owner`, `target_release`, and any other declared input, even when the answer has not changed since the last run an hour ago. Globals make the friction worse, not better, because they are collected up front.

Project-scoped values (the product name, the owning team, the repository slug) are sticky in practice. Document-scoped values (the specific incident ID for an RCA, the decision title for an ADR) are not. The contract should distinguish them and let the host persist the sticky ones.

### Proposal

Add an optional `persistence:` field to each variable declaration in `inputs:` and `globals:`:

```yaml
globals:
  - name: product_name
    kind: string
    required: true
    persistence: project   # ask once per project, reuse across runs
  - name: team_owner
    kind: string
    required: true
    persistence: project
  - name: target_release
    kind: string
    required: false
    persistence: session   # ask once per session, forget after
  - name: incident_id
    kind: string
    required: true
    persistence: none      # ask every run (default)
```

Values are:

* `none` (default): always prompt; never persist. Preserves today's behavior for any consumer that does not opt in.
* `session`: cached for the duration of one host session. Cleared when the session ends.
* `project`: persisted to a host-managed memory file scoped to the current project. Survives across sessions and across document types within the same project.
* `user`: persisted to a user-scoped memory file. Crosses projects. Reserved for genuinely user-level values (display name, preferred Markdown style) and used sparingly.

### Storage contract

The Framework Skill declares intent (`persistence: project`); the host owns mechanism. Recommended host implementation:

* `project` values land in `.copilot-tracking/<host-domain>/variables.yml` (or the host's existing project memory location).
* `user` values land in the host's user memory (e.g., `/memories/fsi-content-vars.md` for Copilot).
* The host MUST surface persisted values for confirmation before reuse, with a one-key-press accept and an explicit override path. Silent reuse is a UX trap.
* The host MUST honor a "clear persisted variables" command per scope.

The storage format is host-owned and not part of the FSI contract. Two hosts may persist values differently; the Framework Skill does not care.

### Schema impact

* The variable declaration shape (used by `inputs:` and `globals:`) gains an optional `persistence` enum: `none | session | project | user`. Default `none`.
* `validate:skills` warns when a variable declares `persistence: user` (encourages authors to think twice).
* `validate:skills` warns when more than ~5 globals declare `persistence: project` (signals a likely UX problem in the Framework Skill, not a hard error).

### Why this matters across document types

The value compounds across the known consumers:

* PRD, ADR, BRD, and RCA in the same project all want `product_name` and `team_owner`. With `persistence: project`, they ask once.
* User-journey templates and engineering-fundamentals briefs reuse the same project context.
* A user authoring three ADRs in one session pays the variable-collection cost once instead of three times.

Without this extension, the variable contract is correct in theory and tedious in practice. With it, the BYOF promise extends to "author once, reuse everywhere in the project."

### Open questions

* Should `persistence: project` interact with the `governance.owners` block (e.g., the team owner persists, the product name does not)? Probably not at the contract layer; let the host inspect both.
* Does `user` persistence introduce a sync problem across machines? Yes, but that is a host concern (the same problem any user memory file already has).
* Should the host be allowed to pre-fill values from project context (`.git`, `package.json`, repository metadata) before falling back to the persisted store? Encouraged but out of scope for the contract.

## Extension 9: Signed output attestations

> Closes a gap surfaced after the initial eight: planner and content-generator outputs have no contract for cryptographic attestation, even though the Sigstore skill is already a built-in.

### Problem

The SSSC Planner, RAI Planner, and Security Planner produce documents that downstream consumers (auditors, release gates, compliance reviewers) need to trust. Today the trust chain is implicit: "the agent ran, the file appeared, the reviewer believed it." There is no machine-verifiable link between the inputs that drove the plan, the Framework Skill version that shaped it, and the resulting artifact.

The `sigstore` framework skill exists in this repo and documents the signing primitives. What is missing is a contract on the Framework Skill side that says: *this item, when produced, MUST be signed; here is what the attestation MUST cover.*

Content-generation outputs (PRDs, ADRs, BRDs, RCAs) face a softer version of the same problem. ADRs in particular are decision records that benefit from "this was the approved version on this date" provenance.

### Proposal

Add an optional `attestation:` block to per-item schemas (initially `document-section`, extensible later) and a paired `signing:` block on the manifest:

```yaml
signing:
  required: true
  method: sigstore-keyless
  identity: https://github.com/<org>/<repo>/.github/workflows/<workflow>.yml@refs/heads/main
  transparency_log: rekor
```

```yaml
- id: success-metrics
  itemKind: document-section
  attestation:
    required: true
    covers:
      - output         # the rendered Markdown/HTML/PDF
      - inputs         # resolved variable values
      - framework_version # manifest version + sha
      - phase          # phase the item was completed in
```

Values for `signing.method`: `sigstore-keyless` (default), `sigstore-key`, `cosign-keyless`, `none`. `none` is legal for draft and prototype use; `validate:skills` warns when `signing.required: true` and `signing.method: none`.

The contract declares *what* must be attested. The Sigstore skill (or an equivalent host-installed signer) declares *how*. A host that does not understand `signing:` MUST fail closed when `signing.required: true` rather than silently emitting unsigned output.

### What the attestation covers

Minimum attestation payload:

* The output artifact (file hash, content type).
* The Framework Skill identifier and manifest sha that produced it.
* The resolved input values (or a hash of them, when inputs contain sensitive data).
* The completion phase and timestamp.
* The `governance.approval` block, if present.

The attestation is itself a binary artifact in the Extension 4 sense: it lives next to the output, the host knows its lifecycle, and `cleanup: ephemeral` does not apply (attestations are durable).

### Storage and verification

* Signed outputs land alongside their attestation file (`<output>.sig`, `<output>.bundle`, or whatever the signer produces).
* The Framework Skill MAY declare a verification recipe under `signing.verify:` that downstream consumers run to confirm provenance. This is documentation, not enforcement.
* `validate:skills` does not invoke the signer (no network, no keys in CI). It validates that `signing:` and `attestation:` blocks are well-formed.

### Schema impact

* Manifest gains an optional `signing` object (method, identity, transparency log, optional `verify` recipe).
* Per-item schemas gain an optional `attestation` object (required flag, `covers` array).
* `validate:skills` gains two lints: `signing` consistency (required + method `none` mismatch), and `attestation.covers` validity (must reference declared inputs and known artifact identifiers).
* No new discovery module changes. `FrameworkSkillDiscovery.psm1` returns the new fields in the manifest payload.

### Why this matters for planners

The SSSC, RAI, and Security planners are the highest-value first consumers because their outputs already need provenance for compliance reasons:

* **SSSC plans** feed Scorecard projections and supply-chain gates. A signed plan binds the gate decision to a specific Framework Skill version and reviewer.
* **RAI plans** feed risk-acceptance reviews. A signed plan is the primary evidence in an RAI audit.
* **Security plans** feed STRIDE remediation backlogs. A signed plan establishes the threat model snapshot the backlog was derived from.

This extension is also the cleanest answer to "why would a planner adopt Framework Skill content-generation extensions at all?" The signing contract is the carrot: planners get cryptographic attestation as a Framework-Skill-declared property instead of as bespoke per-agent code.

### Open questions

* Does `signing.required: true` block local development? Yes by default; recommend a documented `--insecure-skip-signing` host flag scoped to draft Framework Skills only.
* How does signing interact with `phaseMode: revisitable`? Each completion of an item produces a new attestation; the host MUST NOT reuse a stale one when an item is revisited.
* Should `attestation.covers` support custom claim keys? Future work; ship with the fixed set above.
* Cosign vs. native Sigstore: the contract is signer-agnostic. The prototype default is `sigstore-keyless` against the existing GitHub Actions OIDC identity, matching the `sigstore` framework skill already shipped in this repo. Other signers remain valid manifest values; the default is not baked into the schema.

## Cross-cutting concerns

### Backward compatibility

Every proposal is additive. All nine extensions ship as optional fields. A Framework Skill authored against today's schema validates against the extended schema without changes. A host that does not understand the new fields ignores them. The SSSC, RAI, and Security planners are unaffected by default; Extension 9 (signed attestations) is the path by which they may opt in if and when they want Framework-Skill-declared provenance.

### Validation tooling

* `framework-skill-manifest.schema.json` gains `pipeline`, `globals`, `governance`, `phaseMode`, `signing`. All optional.
* New `document-section.schema.json` joins the per-item schema set, including the optional `attestation` block. `validate:skills` dispatches on `itemKind`.
* `validate:skills` gains six lints: variable resolution (Extension 3), binary-artifact existence (Extension 4), governance review currency (Extension 6+8), pipeline kind compatibility (Extension 1, optional), variable persistence sanity (Extension 8), and signing/attestation consistency (Extension 9).

### Discovery module

`FrameworkSkillDiscovery.psm1` does not change. The new fields are returned in the manifest payload it already surfaces. `Resolve-FrameworkSkillPhaseItem` continues to resolve item paths; hosts read the new per-item fields directly.

### Plugin generation

No impact. Plugin generation reads `.github/skills/`, `.github/agents/`, `.github/prompts/`, `.github/instructions/`, `collections/`, and `docs/templates/`. None of these paths are altered by the proposal.

### Existing built-in Framework Skills

No edits required. Nothing in this proposal changes the behavior of the SSSC Planner, the RAI Planner, the Code Review Standards reviewer, or any existing Framework Skill. The extensions are dormant until a content-generation host opts in.

## Out of scope

The following are explicitly **not** part of this proposal. They may become future work but should not contaminate the design discussion of the eight gaps:

* A PRD builder agent itself. The proposal enables one; it does not specify one.
* A `.docx` converter skill. The proposal enables one; the skill is a separate body of work.
* A user-facing UI for collecting variable values. Each host owns its own UX.
* A new `itemKind` for slide decks, reports, dashboards, or other content artifacts. The PRD case validates the pattern; additional kinds are additive future work.
* Conditional logic, loops, partials, or includes in the substitution syntax. Complexity belongs in the renderer skill, not the contract.
* Cross-Framework-Skill composition (one Framework Skill importing items from another). YAGNI.

## Alternatives considered

Three paths address the eight gaps. The proposal selects path C; the other two are documented here so the choice is visible.

### Alternative A: Do nothing

Keep all content-generation logic inside host agents. A PRD builder agent encodes its template structure, variable handling, and pipeline calls in its own prompt. An ADR builder does the same. A BRD builder does the same. BYOF for content generators happens by forking each agent.

* **Pros:** zero schema change, zero tooling change, zero migration. Today's planners and reviewers are unaffected.
* **Cons:** every content-generation host reinvents the same primitives. With at least nine known template types in `docs/templates/`, the cost of "do nothing" is paid nine times. User-authored templates require a fork instead of a drop-in Framework Skill. The asymmetry between "frameworks are data" (planners) and "templates are code" (content generators) becomes a permanent feature of the project.
* **Rejected because:** the BYOF property is the most-cited reason teams adopt HVE Core. Surrendering it for a whole consumer class (one with at least nine identified members) undercuts that pitch.

### Alternative B: Parallel "content bundle" contract

Define a separate manifest schema, discovery module, and validation cmdlet purpose-built for content generation. Leave the FSI untouched. Hosts pick the right contract for their use case.

* **Pros:** the FSI contract stays minimal. Content-generation concepts (pipelines, variables, binary artifacts) live in their own schema where they cannot pollute the planner mental model.
* **Cons:** two parallel contracts double the surface area for documentation, validation, plugin generation, and authoring guidance. BYOF authors learn one shape for security frameworks and a second shape for content templates. Discovery and override semantics get re-implemented and inevitably drift.
* **Rejected because:** the eight gaps are additive on top of the FSI, not orthogonal to it. Phase walking, draft quarantine, override semantics, and authoring loop all transfer cleanly. A second contract pays a permanent cost to avoid a one-time additive change.

### Alternative C: Additive FSI extensions (this proposal)

Extend the FSI with optional fields (`pipeline`, `globals`, `governance`, `phaseMode`) and recognize a new `itemKind: document-section` with its own per-item schema. Existing Framework Skills validate unchanged. Existing hosts ignore the new fields.

* **Pros:** one contract, one schema, one discovery module, one authoring loop. BYOF parity extends to the new consumer class. Existing planners and reviewers gain nothing and lose nothing.
* **Cons:** ~30% growth in the manifest schema (four optional fields), one new per-item schema, four new validation lints. Maintenance surface increases proportionally.
* **Selected because:** the cost is bounded and one-time; the benefit compounds with every new content-generation host. The design constraints (additive only, host owns meaning, no silent shadowing) keep the growth from contaminating the existing contract.

## Effort and risk

Each extension is sized in T-shirt terms (S = days, M = a week, L = multi-week). Risks are the specific things most likely to derail the work, not generic caveats.

| Extension                       | Size | Primary risk                                                                                                                            | Mitigation                                                                                               |
|---------------------------------|------|-----------------------------------------------------------------------------------------------------------------------------------------|----------------------------------------------------------------------------------------------------------|
| 1. Pipeline manifests           | M    | Strict vs. loose `consumes`/`produces` typing is a forcing function for every future stage author; getting it wrong fragments adapters. | Land strict typing first; relax to loose only on demonstrated need.                                      |
| 2. `itemKind: document-section` | M    | Per-item schema design locks in vocabulary that later content kinds (slides, dashboards) inherit by analogy.                            | Validate the shape against at least one non-PRD content kind before promoting from `draft`.              |
| 3. Variable substitution        | S    | Substitution syntax choice (`{{var}}`) is irreversible once Framework Skills ship.                                                      | Pick the convention that already dominates prompt templates in the repo; document the choice.            |
| 4. Binary artifact contract     | S    | Working-directory convention leaks across hosts that adopt it; stale artifacts accumulate.                                              | Ship with ephemeral default and an explicit cleanup contract documented for hosts.                       |
| 5. First-class `selectWhen`     | S    | Generalizing to `control` and `rule` later is cheap; restricting now is reversible.                                                     | Ship as `document-section`-only; revisit when a second host requests it.                                 |
| 6+8. Governance metadata block  | M    | Built-ins need backfill if `governance` becomes required at promotion; back-pressure on Framework Skill authors.                        | Keep optional in the schema; warn-only in `validate:skills`; require only at host-level opt-in.          |
| 7. Non-linear phase mode        | S    | Risks leaking host UX semantics into the Framework Skill contract.                                                                      | Defer until a second host (DT method coach is the candidate) requests it.                                |
| 8. Variable persistence         | S    | Silent reuse of persisted values is a UX trap; users lose track of where a value came from.                                             | Require host to confirm persisted values before reuse; provide explicit "clear" command.                 |
| 9. Signed output attestations   | M    | Signing pipeline failures block planner output; key/identity management is non-trivial in CI.                                           | Default to `sigstore-keyless` against existing CI identities; ship `--insecure-skip-signing` for drafts. |

Total sizing if every extension lands: roughly 5–6 weeks of focused work plus prototype validation. The prototype (Extensions 2+3, ~1.5 weeks) remains the right first commit because it de-risks the per-item schema before the rest depends on it, but the work that follows is no longer speculative: it is amortized across the existing template inventory.

**Strategic risks worth naming:**

* **Per-item schema lock-in.** Once ADR, BRD, and RCA Framework Skills ship against `document-section`, the schema becomes hard to change. Mitigated by validating the prototype against at least one non-PRD template (recommend ADR) before promoting Extension 2 from `draft`.
* **Migration drag.** Each existing template in `docs/templates/` is a candidate, but each migration is also a coordination cost with the agents that consume it. Sequence migrations behind owner agreement; do not bulk-convert.
* **Governance gap during transition.** Templates that have not yet migrated still rely on free-form Markdown discipline. Extension 6+8 (`governance:` block) only helps Framework Skills that have migrated. Document the transition explicitly in the BYOF guide so authors know which templates are FSI-backed.

## Decisions needed

Before any of this lands, four decisions need explicit answers:

| ID    | Decision                                                                                        | Why it matters                                                                                                       |
|-------|-------------------------------------------------------------------------------------------------|----------------------------------------------------------------------------------------------------------------------|
| DN-01 | Do we accept `pipeline:` as a manifest-level concept at all?                                    | If no, all content-generation chaining stays in host code and Extension 1 collapses.                                 |
| DN-02 | Do we standardize substitution syntax (`{{var}}`) at the contract layer?                        | If no, every renderer skill carries its own templating and BYOF interoperability suffers.                            |
| DN-03 | Is `governance:` mandatory for new Framework Skills, optional always, or required at promotion? | Drives whether `validate:skills` warns or errors and whether built-ins need backfill.                                |
| DN-04 | Do we ship `selectWhen` as `document-section`-only or generalize to all `itemKind`s?            | Generalization is cheap now and expensive later; restricting is safer if we are unsure of the cross-domain semantic. |

Each decision is independent. None blocks the others; the design space partitions cleanly.

## Suggested next steps

1. **Discuss this proposal.** Confirm the eight gaps are the right gaps and the four design constraints are the right constraints.
2. **Resolve DN-01 through DN-04.** Each decision unblocks one or more extensions.
3. **Prototype Extension 2 and Extension 3 against a throwaway PRD Framework Skill.** They have the highest leverage and the smallest blast radius.
4. **Land Extensions 1, 4, 5, 6+8 once Extension 2 is stable.** They depend on the per-item schema being settled.
5. **Defer Extension 7** until a second host besides PRD builder needs non-linear phases.

A successful prototype is one where a user-authored PRD template Framework Skill, dropped into `.copilot-tracking/framework-imports/product-planning/`, is discovered, validated, rendered, and converted to `.docx` by a generic content-generation host with zero agent code changes.

That outcome is the same property Framework Skills already give the SSSC Planner today: BYOF parity between built-ins and user authorship. The extensions in this document exist to extend that property to a new consumer class without disturbing the existing one.
