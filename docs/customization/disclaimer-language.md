## Why Centralized

HVE Core planner agents (RAI, Security, SSSC, Accessibility, Sustainability) all generate assessments that require review by a qualified human reviewer. Each planner displays a CAUTION block during startup and folds the same language into handoff artifacts. Centralizing this language has three benefits:

* Legal and compliance teams review a single source rather than chasing duplicates across agent files.
* Wording stays consistent across planners, so reviewers see the same "this is an assistive tool" framing regardless of which planner produced an artifact.
* CI validation has a single file to read for Tier 2 disclaimer checks rather than parsing every agent file.

## Where It Lives

Three files own the disclaimer surface. Each has a distinct role.

| File                                                              | Role                                                                                                                                  |
|-------------------------------------------------------------------|---------------------------------------------------------------------------------------------------------------------------------------|
| `.github/instructions/shared/disclaimer-language.instructions.md` | Authoritative source for the per-planner CAUTION blocks. Agents reference sections via `#file:` links.                                |
| `.github/config/disclaimers.yml`                                  | Tier 2 full disclaimer text per planner, the artifacts each disclaimer applies to, and stable ids used by validation.                 |
| `.github/config/footer-with-review.yml`                           | Tier 1 AI-content note, the human review checkbox text, and the artifact-classification rules that map artifacts to required footers. |

`scripts/linting/Validate-PlannerArtifacts.ps1` reads `disclaimers.yml` and `footer-with-review.yml` as the single source of truth when checking generated artifacts. Update those YAML files (not the agents) when wording changes.

## How Planners Source It

Planner identity instructions and handoff phases reference the shared file rather than inlining the text. The pattern is:

1. The agent's identity file declares a "Disclaimer and Attribution Protocol" step.
2. That step references `.github/instructions/shared/disclaimer-language.instructions.md` and the planner-specific section heading (for example "RAI Planning" or "SSSC Planning").
3. The agent displays the CAUTION block verbatim before any phase work begins.
4. Handoff phases read the matching entry in `.github/config/disclaimers.yml` for the artifact being generated and embed the full disclaimer text in the artifact footer.
5. Tier 1 artifacts read the AI-content note and human-review checkbox from `.github/config/footer-with-review.yml`.

The Sustainability planner adds an out-of-band disclosure refusal in the same shared file. Requests to draft regulator-facing filings (CSRD, ESRS, SEC climate, GHG Protocol, TCFD, ISO 14064/14067) are routed to qualified counsel; the planner refuses to generate filing text or attestation language.

## What NOT to Put in Framework Skill Manifests

The `metadata.disclaimer` field on a Framework Skill manifest is reserved for **framework-specific technical caveats**. Examples that belong there:

* "SCI outputs are directional estimates, not an audited carbon disclosure."
* "SLSA level claims are self-attested unless verified by a third-party auditor."
* "WCAG conformance scores reflect automated checks only; manual audit required for AA or AAA claims."

Examples that do NOT belong in a Framework Skill manifest:

* The full RAI, Security, SSSC, Accessibility, or Sustainability planner CAUTION block.
* Generic "this content was generated by AI and requires human review" boilerplate.
* Verbatim license text (the validator already aggregates `attributionText` into the repo-root `THIRD-PARTY-NOTICES` file).

When a per-item document-section template needs to override the manifest-level
technical caveat (for example marking one Privacy Impact section with
`severity: warning`), use the per-item `disclaimer:` field and keep the override
scoped to that section. Domain-level legal review boilerplate is always sourced
from `.github/instructions/shared/disclaimer-language.instructions.md`. The
[`framework-skill-interface` skill](../../.github/skills/shared/framework-skill-interface/SKILL.md)
documents the contract.

## Review and Contribution

The disclaimer files are owned by the hve-core maintainers in coordination with legal and compliance reviewers. To propose a change:

1. Open a pull request that updates the relevant section of `.github/instructions/shared/disclaimer-language.instructions.md` and the matching entry in `.github/config/disclaimers.yml`.
2. Include the rationale (regulatory update, scope change, new planner) in the pull request description.
3. Tag the planner-specific maintainers and the legal review contact listed in `CODEOWNERS`.
4. Wait for `npm run lint:all` to pass. The `Validate-PlannerArtifacts.ps1` check confirms agents reference the new wording correctly and that `disclaimers.yml` ids resolve.
5. After merge, planner agents pick up the new wording on their next session because they reference the shared file rather than embedding the text.

Review cadence is at least annual, or whenever a planner's scope materially changes (new framework, new artifact type, new regulatory regime).
